<objective>
Refactor the provider layer from flat provider_order config to a per-tier provider_chain with (provider, model) pairs and implement multi-provider support.

Purpose: The existing provider code hard-wires provider selection through a single AI_INVESTING_PROVIDER env var and flat env_model_keys. This plan upgrades to a chain-based architecture where each tier can independently specify an ordered list of (provider, model) fallback pairs, supporting OpenAI, Anthropic, Google Gemini, Groq, and any OpenAI-compatible endpoint.
Output: Refactored config schema, provider resolution logic, 4 dedicated adapters + 1 generic adapter, backward-compatible settings, and regression tests.
</objective>

<requirement_traceability>
Parent requirements: `PROV-02`, `PROV-03`, `PROV-05`
Slice acceptance:
1. `model_profiles.yaml` uses `provider_chain` with `(provider, model)` pairs per tier.
2. `AppContext.get_provider()` resolves through the chain, skipping entries without valid credentials.
3. Dedicated adapters for OpenAI, Anthropic, Google Gemini, Groq are installable via optional extras.
4. A generic `openai_compatible` adapter works with any OpenAI-compatible endpoint via `OPENAI_COMPATIBLE_BASE_URL`.
5. `AI_INVESTING_ALLOW_FAKE_FALLBACK` gates fake provider fallback in production.
6. Missing keys for a non-fake provider produce clear validation errors.
7. All existing tests pass unchanged with the fake provider.
Out of scope: Structured logging, token tracking persistence, retry logic (Plan 02).
</requirement_traceability>
