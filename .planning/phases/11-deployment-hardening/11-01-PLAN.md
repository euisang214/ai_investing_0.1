# Plan 11-01: Multi-Stage Dockerfile, Health Probes, Profiles, and Credential Safety

## Goal
Produce a production-worthy Docker setup: lean multi-stage image, health/ready probes, dev/prod profiles, and credential safety enforcement. Existing dev workflow must remain unchanged.

## must_haves
- Multi-stage Dockerfile with `base`, `prod`, and `dev` targets; prod image excludes tests, examples, docs, n8n, scripts, and dev deps
- `GET /health` returns 200 trivially; `GET /ready` returns 200 only when DB is reachable (503 otherwise)
- Health/ready endpoints are exempt from API key authentication
- `docker compose --profile dev` starts dev stack; `docker compose --profile prod` starts prod stack; default (no profile) behaves like current dev setup
- Production profile reads DB credentials from environment variables and refuses to start with default `postgres:postgres`
- Alembic migrations run cleanly in both profiles
- All existing tests pass without modification

## Tasks

<task id="1" title="Multi-Stage Dockerfile">
### Description
Restructure the existing Dockerfile into a multi-stage build with `base`, `prod`, and `dev` targets.

### Steps
1. Rewrite `Dockerfile` with three stages:
   - **`base`** — `FROM python:3.11-slim AS base`
     - Set `PYTHONDONTWRITEBYTECODE=1`, `PYTHONUNBUFFERED=1`
     - Set `WORKDIR /app`
     - Install system deps (`curl`, `build-essential`), install `uv`
   - **`prod`** — `FROM base AS prod`
     - `COPY pyproject.toml README.md /app/`
     - `COPY alembic.ini /app/`
     - `COPY alembic /app/alembic`
     - `COPY src /app/src`
     - `COPY config /app/config`
     - `COPY prompts /app/prompts`
     - `RUN uv pip install --system .` (runtime deps only, NOT editable, NOT `.[dev]`)
     - `CMD ["uvicorn", "ai_investing.api.main:app", "--host", "0.0.0.0", "--port", "8000"]`
   - **`dev`** — `FROM base AS dev`
     - Copy everything: `pyproject.toml`, `README.md`, `alembic.ini`, `alembic/`, `src/`, `config/`, `prompts/`, `examples/`, `docs/`, `n8n/`, `scripts/`, `tests/`
     - `RUN uv pip install --system -e ".[dev]"` (editable with dev deps)
     - Same `CMD` as prod

2. Verify the prod target builds:
   - `docker compose build --build-arg BUILDKIT_INLINE_CACHE=1`

### Acceptance
- `docker build --target prod .` completes successfully
- `docker build --target dev .` completes successfully  
- Prod image does NOT contain `/app/tests`, `/app/examples`, `/app/docs`, `/app/n8n`, `/app/scripts`
- Dev image DOES contain all those directories
</task>

<task id="2" title="Health and Readiness Endpoints">
### Description
Add `/health` and `/ready` endpoints to the API, exempt from authentication.

### Steps
1. Add a `ping()` method to `Database` in `src/ai_investing/persistence/db.py`:
   ```python
   def ping(self) -> bool:
       """Return True if the database is reachable."""
       try:
           with self.engine.connect() as conn:
               conn.execute(text("SELECT 1"))
           return True
       except Exception:
           return False
   ```
   Import `text` from `sqlalchemy`.

2. Update `ApiKeyMiddleware.dispatch` in `src/ai_investing/api/security.py`:
   - Add a path-based bypass: if `request.url.path` is `/health` or `/ready`, set `request.state.role = "operator"` and call `call_next(request)` without checking the API key.
   - This goes at the TOP of `dispatch()`, before the `self._active` check.

3. Add endpoints in `src/ai_investing/api/main.py` inside `create_app()`:
   ```python
   @app.get("/health")
   def health() -> JSONResponse:
       return JSONResponse(content={"status": "ok"})

   @app.get("/ready")
   def ready(request: Request) -> JSONResponse:
       db = _context(request).database
       if db.ping():
           return JSONResponse(content={
               "status": "ready",
               "checks": {"database": "ok"},
           })
       return JSONResponse(
           status_code=503,
           content={
               "status": "not_ready",
               "checks": {"database": "unreachable"},
           },
       )
   ```

### Acceptance
- `GET /health` returns `{"status": "ok"}` with 200
- `GET /ready` returns `{"status": "ready", "checks": {"database": "ok"}}` with 200 when DB is up
- `GET /ready` returns 503 with `not_ready` when DB is unreachable
- Both work without an API key when `auth_enabled=true` and `api_keys` is configured
</task>

<task id="4" title="Credential Safety on Startup">
### Description  
Refuse to start the API when auth is enabled but default insecure DB credentials are detected.

### Steps
1. Add a `validate_production_settings()` method or standalone function in `src/ai_investing/application/context.py`:
   ```python
   def _validate_production_settings(settings: Settings) -> None:
       """Raise SystemExit if production config uses insecure defaults."""
       if not settings.auth_enabled:
           return  # dev mode, skip validation
       if "postgres:postgres@" in settings.database_url:
           raise SystemExit(
               "FATAL: Default database credentials detected in production mode. "
               "Set AI_INVESTING_DATABASE_URL with secure credentials, "
               "or set AI_INVESTING_AUTH_ENABLED=false for development."
           )
   ```

2. Call `_validate_production_settings(resolved_settings)` at the start of `AppContext.load()`, after resolving settings but before connecting to anything.

3. Ensure existing tests pass — they all use `auth_enabled=False` or don't set `api_keys` (which disables auth), so they'll never hit this check.

### Acceptance
- App starts normally with `auth_enabled=false` and `postgres:postgres`
- App starts normally with `auth_enabled=true` and non-default credentials
- App refuses to start (SystemExit) with `auth_enabled=true` and `postgres:postgres@` in DB URL
- All existing tests pass unchanged
</task>
