# Plan 10-01: API Key Auth, CORS, and Role-Based Access

## Goal
Protect all API endpoints with API key authentication, add CORS configuration, and enforce operator-only access on mutation endpoints. Existing tests must remain unaffected.

## must_haves
- Every endpoint returns 401 without a valid X-API-Key header when auth is enabled
- AI_INVESTING_API_KEYS env var configures accepted keys with roles
- CORS defaults to blocking cross-origin requests, configurable via AI_INVESTING_DOMAIN
- Operator-only endpoints return 403 for readonly keys
- Existing tests pass without modification (auth bypass via AI_INVESTING_AUTH_ENABLED=false)

## Tasks

<task id="1" title="Settings and Security Module">
### Description
Add auth and CORS settings to Settings, create the security module.

### Steps
1. Add to `src/ai_investing/settings.py`:
   - `auth_enabled: bool = True` — controls whether auth middleware is active
   - `api_keys: str = ""` — comma-separated `key:role` pairs (e.g. `sk-abc:operator,sk-xyz:readonly`)
   - `domain: str = ""` — comma-separated allowed CORS origins

2. Create `src/ai_investing/api/security.py` with:
   - `parse_api_keys(raw: str) -> dict[str, str]` — parses the `key:role` format, returns `{key: role}` dict
   - `ApiKeyMiddleware` — Starlette BaseHTTPMiddleware that:
     - If `auth_enabled=false`, skips checking and sets `request.state.role = "operator"` (full access)
     - If `auth_enabled=true`, reads `X-API-Key` header, looks up in parsed keys dict
     - Missing/invalid key → 401 JSON response using `_error_response` pattern: `{"error": {"code": "unauthorized", "message": "Missing or invalid API key"}}`
     - Valid key → sets `request.state.role` to the key's role, continues
   - `require_role(required_role: str) -> Depends` — FastAPI dependency that:
     - Reads `request.state.role`
     - If role doesn't match required_role → 403 JSON response: `{"error": {"code": "forbidden", "message": "This endpoint requires operator role"}}`
     - If role matches → passes through

### Acceptance
- `parse_api_keys("sk-a:operator,sk-b:readonly")` returns `{"sk-a": "operator", "sk-b": "readonly"}`
- Middleware sets `request.state.role` correctly
- 401 for missing key, 403 for wrong role
</task>

<task id="2" title="Wire Auth and CORS into FastAPI App">
### Description
Integrate the security middleware and CORS middleware into `create_app()`, and apply `require_role("operator")` to all operator-only endpoints.

### Steps
1. In `src/ai_investing/api/main.py`:
   - Import `ApiKeyMiddleware`, `require_role` from `api.security`
   - Import `CORSMiddleware` from `fastapi.middleware.cors`
   - After creating the `app` object, add middleware in order:
     a. `ApiKeyMiddleware` — uses `settings.auth_enabled` and parsed `settings.api_keys`
     b. `CORSMiddleware` — if `settings.domain` is non-empty, parse as comma-separated origins list and configure CORS with:
        - `allow_origins` = parsed origins
        - `allow_methods` = `["GET", "POST", "PUT", "DELETE", "OPTIONS"]`
        - `allow_headers` = `["X-API-Key", "Content-Type"]`
        - `allow_credentials` = `False`
     c. If `settings.domain` is empty, skip adding CORSMiddleware entirely (default-deny)

2. Add `require_role("operator")` as a FastAPI `Depends()` parameter to every operator-only endpoint (25 endpoints total, per CONTEXT.md classification):
   - All POST/DELETE endpoints that trigger mutations, LLM runs, or side effects
   - Example: `def create_coverage(payload: CoverageCreateRequest, request: Request, _=Depends(require_role("operator")))`

3. Readonly endpoints (14) get NO `require_role` dependency — any authenticated user can access them

4. Access `settings` by reading it from `app.state.context.settings` in the middleware init within the `lifespan()` or by passing settings to `create_app()`.

### Acceptance
- `create_app()` adds ApiKeyMiddleware and conditionally adds CORSMiddleware
- All 25 operator endpoints have `require_role("operator")` dependency
- All 14 readonly endpoints work without role check (any valid key)
</task>

<task id="4" title="Comprehensive Security Tests">
### Description
Write tests for auth, CORS, and role-based access control.

### Steps
1. Create `tests/test_api_security.py` with test classes:

   **TestAuthBypass:**
   - `test_auth_disabled_allows_all_requests` — with `auth_enabled=false`, requests without API keys succeed (200)
   - `test_default_role_is_operator_when_auth_disabled` — with auth disabled, even operator-only endpoints work (no 403)

   **TestAuthMiddleware:**
   - `test_missing_api_key_returns_401` — no `X-API-Key` header → 401 with correct error shape
   - `test_invalid_api_key_returns_401` — bogus key → 401
   - `test_valid_api_key_passes_through` — valid key → 200 on GET endpoint
   - `test_error_response_format` — 401 body matches `{"error": {"code": "unauthorized", "message": "..."}}`

   **TestRoleBasedAccess:**
   - `test_operator_key_accesses_operator_endpoint` — operator key → 200 on POST /coverage
   - `test_readonly_key_blocked_from_operator_endpoint` — readonly key → 403 on POST /coverage
   - `test_readonly_key_accesses_readonly_endpoint` — readonly key → 200 on GET /coverage
   - `test_operator_key_accesses_readonly_endpoint` — operator key → 200 on GET /coverage
   - `test_all_operator_endpoints_require_role` — iterate all operator-only route paths, verify each returns 403 with a readonly key

   **TestCORS:**
   - `test_no_cors_headers_when_domain_empty` — default: no CORS headers in response
   - `test_cors_headers_present_when_domain_set` — with `AI_INVESTING_DOMAIN=http://localhost:3000`, preflight OPTIONS returns correct headers
   - `test_cors_allows_configured_origin` — actual request with `Origin` header from allowed domain gets CORS headers

   **TestParseApiKeys:**
   - `test_parse_valid_keys` — parses `key1:operator,key2:readonly`
   - `test_parse_empty_string` — returns empty dict
   - `test_parse_single_key` — single key works

2. Create test helpers:
   - `_create_auth_app(auth_enabled, api_keys, domain)` — creates an app with specific security settings
   - Use `TestClient` with custom headers: `client.get("/coverage", headers={"X-API-Key": "sk-test"})`

3. Ensure existing tests in `tests/test_api.py` continue to pass:
   - The `context` fixture creates `Settings(...)` without `auth_enabled` → defaults to `True`
   - BUT: the fixture also doesn't set `api_keys` → empty string → no valid keys
   - Solution: the conftest `context` fixture should set `auth_enabled=False` to preserve existing behavior
   - Alternatively, `create_app(context)` when `api_keys` is empty could implicitly disable auth

   **Decision:** When `api_keys` is empty AND `auth_enabled` is not explicitly set, treat auth as disabled. This means:
   - Production: must set `AI_INVESTING_API_KEYS` — if empty and `auth_enabled=true`, log a warning and pass all requests (no keys to check against, so nothing to block)
   - OR: simpler approach — if `api_keys` is empty, skip auth entirely regardless of `auth_enabled`. This is safest for backward compatibility.

### Acceptance
- 15+ security tests pass
- All existing test_api.py tests pass without modification
- Zero regressions in full test suite
</task>
