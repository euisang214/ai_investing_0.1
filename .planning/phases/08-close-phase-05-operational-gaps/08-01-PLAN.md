  <name>Task 1: Fix worker running-state truthfulness</name>
  <files>src/ai_investing/application/services.py, tests/test_worker_runtime.py</files>
  <action>In `AnalysisService.execute_refresh_job()`, add a call to `Repository.start_refresh_job(job_id, run_id=..., worker_id=worker_id)` before delegating to `refresh_company()`. This requires generating or resolving the `run_id` first so both `start_refresh_job` and `refresh_company` reference the same run. The key constraint is that `refresh_company` creates the run — so the approach should either:

Option A: Call `start_refresh_job` with the job_id to mark it `RUNNING` before calling `refresh_company`, then use the returned job's run_id assignment. Since `refresh_company` creates a new run internally, `start_refresh_job` should mark the job as RUNNING without requiring a `run_id` yet (use a transitional state).

Option B: Alternatively, pass the job_id through to `refresh_company` and have `_sync_operational_state` call `start_refresh_job` once the run is created and the run_id is known.

Choose the approach that minimizes disruption. Add a regression in `tests/test_worker_runtime.py` that asserts the job transitions to `RUNNING` status before the run completes.</action>
